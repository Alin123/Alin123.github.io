iOS多线程总结

发表于 2016-08-20   |   分类于 学习

交付队列与交付线程

iOS实现多线程编程的三种方法：

  1. NSThread：显示的(根据要执行的操作)创建线程
  2. NSOpreation、NSOperationQueue：通过将任务封装成operation、创建和使用queue的方式实现多线程编程
  3. GCD：将任务放在block中、创建和使用dispatch_queue_t的方式实现多线程编程

后两者抽象层度更高(不显示的创建线程却可以实现多线程编程)，它们的相似点也很多：任务都被封装，都是通过交付队列的方式开始执行。任务真正被执行是在线程中，我们只是将任务交付给队列，队列帮我们选择或创建线程去执行他们。也可以将操作队列简单的理解成一种先进先出的数据结构。

在GCD中实现并行与串行是将任务交付给对应的并行与串行队列，在NSOperationQueue中可以通过添加依赖或设置最大并发数实现，
[[NSOperationQueue alloc] init]
返回GCD中的并行队列dispatch_queue_create("com.alin.multithread.concurrentQueue", DISPATCH_QUEUE_CONCURRENT)。并行队列将先入对的任务交付线程执行，然后接着交付下一个，而不管前一个是否完成(通常情况下…)。串行队列可以理解成等待前一个执行完成后再交付下一个任务开始执行。银行的都个窗口就像多个串行多列，先拿到号码的可以先办理任务，下一个人必须等待前一个完成。并行队列就像大家排队领申请表，领完后去一边填写，谁先填写完交上去的顺序不定，但领表的循序是先来先领的。

异步与同步

同步：就是在发出一个功能调用时，在没有得到结果之前，该调用就不返回。

异步：异步的概念和同步相对。当一个异步过程调用发出后，调用者不会立刻得到结果。实际处理这个调用的部件是在调用发出后，通过状态、通知来通知调用者，或通过回调函数处理这个结果。

开始下载：其中[self imageUrlStringArray]返回一个数组包含四个图片的URL字符串，[self presentImageViewArray]包含四个对应的imageView。

将任务提交给队列：开始执行初始化NSBlockOperation用到的block时，进入了线程(虽然我们没有显示的创建一个子线程)。现在完成后需要切换到主线程加载image。从结果来看dispatch_async(dispatch_get_main_queue(), <^(void)block>)和[[NSOperationQueue mainQueue] addOperation:[NSBlockOperation blockOperationWithBlock:<^(void)block>]]效果一样。修改信号量的操作self.finishedSign ++应该和接下来的判断保持原子性，幸好它们是在主串行队列，其他任务会等待到判断if (self.finishedSign >= 4)和附值self.imageLoading = NO完成后再继续执行。因此不会造成“多线程买票”出现票数为负(不加锁)的异常情况。

用任务任务组的形式也可以实现“信号量”的效果，在四个下载任务完成后更改imageLoading标识量。

生命如歌－礼物

发表于 2016-08-13

8月1日，下午5点18分。“新公司”的工作有了阶段性的小结：版本开发完毕，测试的童鞋正在测试中…

距离上一次写点东西已有一个月的时间了。这段时间都忙了什么呢？找工作是其中之一，最重要的要数某位童鞋的生日了O(∩_∩)O。生日对于一个人的重要性不言而喻，它时间为刻度记录了我们的成长，提醒我们明天是新的一个人生阶段的开始…

还记得是在第一次见你的时候，问了你的生日：0624。你说想要一个和你等身高的布偶(哦，最好不是小熊)，还讲了关于这个礼物背后的故事，它是你小时候的非常渴望得道的礼物，是儿时的一个梦想(估计那时候每到生日的时候你都希望有个一等身高的布偶走到你的身边，让后亲口对你说：“生日快乐”)。为此你还跟自己的父母吵了架。我虽然不知道当时的情形，但却完全能体会到你当时的感受…

也许每个人小时候都有一件非常渴望得道的东西，天上的一个星星或是印有可爱小兔图案的文具盒。这个愿望就一直潜伏在我们内心深处，时不时的伸出手拨弄我们的心弦。

一个崭新的属于我自己的书桌。我从小学到初中，乃至高中(虽然那时候都用学校的书桌了)一直都希望自己也有一个。然而，一直是我堂哥那个锢了三圈铁条、你一动它就跟着动的书桌。也曾向父母表达过自己的想法，但结果都是一样的…不记得是哪年夏天的一天，我哥我姐不上学了他们的书桌不用了。我想即使不能换个新的，换个自己亲哥亲姐的、好一点的总该木有问题吧！然而当我再一次向父母表达的想法时，却是又一次的回绝(当时是父母已经答应把书桌给我三姨家的两个孩子了)…当时也是各种期待、争辩、痛哭、沉默。以至于到后来我都希望自己是个哑巴了，那样就不用跟他们争辩了，也不希望他们了解那个潜伏在内心的愿望了。

…

时间就这样慢慢的走呀走，我的那个儿时的愿望也抱着时间的尾巴，消失于内心深处。只是突然听到你的故事让我仿佛回到了小时候那个追逐一个书桌梦想的时代…我暗自下定决心，一定要给你买一个。能实现你的梦想比我得道那个书桌还要让我期待，我是那么的爱你～

终于，一个紫色的小熊还是陪在了你的身边，虽然期间有些不愉快。亲爱的我是那么的爱你，突然发现已渐渐离不开你，无时无刻的想念让我痛恨这空间上的距离。我羡慕那只小熊，羡慕它能此刻能陪在你的身边，羡慕它总是那么称你的心意。我在想，假如有一天我们也天天在一起，我是不是也是你心中的那个“等高的布偶”…

生命如歌－居安思危

发表于 2016-08-01   |   分类于 生活

16年6月28日，下午4点46分。离开上家公司正好一个星期。

公司因为种种原因吧，决定不再发展与预约美甲服务相关的业务，导致我们产品技术中心裁掉了一大批人，当然也包括我这个移动端的iOS开发工程师。想想就在不久之前，公司还一起出去团建，一起出去各种嗨！期间的会议上，老板还各种长篇大论、各种高谈阔论…而今却以是到了裁员以缩减开支的境地(当然这每个人都有责任的)。想想当初自己也是满怀期待的来到公司，希望跟着公司一起成长，有个长足稳定的工作！现在开来这种想法是完全错误的。

任何时候我们都不能认为自己安稳了，要时刻有危机意识，要居安思危。一旦我们感觉安稳了，危机就会悄悄潜伏着并在那里积蓄力量。安稳会让我进步缓慢，停滞不前，甚至退步。拿我来说，我认为我是一个认真负责的员工，但那时候眼前的一切让我进步缓慢。工作中只专注于解决公司APP中的一系列问题，没有更深入的了解自己领域相关的知识。仅仅局限会用，能解决问题现在看来还远远不够。换句话说做几个APP不重要，重要的是能否对自己所处的领域有深刻的了解。对iOS开发来说，没有深刻的了解，就写不出一个优秀的第三方，在APP性能优化方面就会显得手足无措。徘徊于初级与中级开发工程师之间。

从另一方面来说，这次离职也是一件好事。温水煮青蛙多不好吖！是时候重新收拾行囊再出发了…

一周的时间过去了。写简历，改简历，期间也面试了两家，到目前为止一个不合适，另一个也没了消息，估计也不了了之了。不禁想起了刚来北京找工作的时候也是面试了好多家，大概持续了三周，最终功夫不负有心人，工作机会总算向我伸出了橄榄枝。现在，生活仍得继续，工作还得找。

对于将来的计划：对iOS有深入的探索，学习视频音频相关的技术；年内入门HRML5；掌握React Native开发技术。

在职场上没有所谓的稳定，只有不断的强大自己才是王道，才能这个瞬息万变的时代立于不败之地。

发表于 2016-06-07   |   分类于 生活

6月7日 晴
晚上下了一阵大雨。今天早上，清爽的空气又一次滋润了帝都北京。
上班的路上，看到潮湿的地面，低洼的地方还会有一汪水。突然想到小时候听到的一首歌”快点告诉你”，于是根据前两句歌词”下了一整夜的雨，早起就是好天气”上网搜了一下，原来是动画城的主题歌，怪不得虽然没有特意听过这首歌却也能哼的上来。
说到动画城让我想起可一部极具想象力的动画片”马丁的早晨”，主人公马丁每天早上醒来都会扮演一个全新的、光怪陆离的(也许在当时的我看来那是真的会发生在在生活中的事情)角色，然后完成一次惊险、刺激的冒险…
对于一个向来语文很糟糕的人来说写一篇800字的作文来说简直是对大脑造成各种真实伤害。现在想来原因大概有三个：

  1. 读书太少。对一个理工男来说也是只会对像”坏蛋是怎么炼成的”这样的小说感兴趣，而四书五经在当时读来就如同嚼蜡，看鲁迅先生的文章也是像看天书一样。
  2. 没有足够的阅历。对于一个直到上大学才出过远门的人来说，行万里路在金钱和时间的维度上都是不可能实现的。见到的人也大多是亲戚、朋友或是师长，当时他们也与在忙生计的事情，或者是我并没有在意他们讲述的故事。就这样阅历的局限性在所难免了。
  3. 想象力严重缺乏。想象力缺乏就是硬伤了。哦！也许不全是。多读书、多见识一些人和事就会丰富我们的想象力了呢！到那时，见到一个帽子像形状的图案，我们的第一反应也许不再是一顶帽子，而是一条吞了大象的蟒蛇呢！

说到想象力，现在想来小时候看的最具想象力的动画片就是”马丁的早晨”，每天一个新的角色和一场全新的冒险正是多少小朋友正在经历的(或是期盼的)生活吖！就在码字的同时我回顾了这部小时侯的记忆，在现在看来它依然那么充满想象力。羡慕他每天都有一个全新的生活，羡慕他有一个爱她的罗娜(不过我有我的宝贝！)…但童话终究是童话，童话中的生活我们无非复制，童话能给我们的也许是对待生活态度，就像马丁那样，每天都是新的一天，我们都能学到新的东西，对生活都有不同的感悟，不是么！

Xcode使用CocoaPods管理第三方类库时的异常

发表于 2016-06-01   |   分类于 Study

场景描述：

2016年5月31日下午5点58分，项目组决定重启商家版UI重构版(之前该版本为能直接上线且直接在master主分支上开发的)。于是:

  1. 新建名为Seller_New_UI文件夹，并通过命令行进入该文件夹；
  2. 将代码克隆到该文件夹：git clone http://gitlab/10.100.10.1/myProject.git
  3. 进入工程文件：cd myProject
  4. 查看所有分支：git branch -a，发现当前在master主分支，本地和远端都木有“UI重构版分支”
  5. 基于master主分支创建“UI重构版分支”分支：git branch feature/dev_newUI_v2.0
  6. 查看所有分支：git branch -a，发现本地已经有了feature/dev_newUI_v2.0分支
  7. 将新建分支同步到远端：git push origin master
  8. 使用git branch -r查看远端所有分支，发现已经有啦
  9. 切换到本地新建分支：git checkout feature/dev_newUI_2.0

实际问题

接下来就是打开工程文件myProject.xcworkspace(使用了CocoaPods管理了第三方类库)，编译…
问题来了，Xcode抛出异常：

解决方法

  1. 接着使用命令行键入pod命令：pod install
  2. 待命令执行完成，重新打开myProject.xcworkspace工程文件，编译…

这时候编译就成功啦！(大都数情况是失败的，这时候原因大多是该引用的文件未添加应用，已经删除的文件引用还没删除，证书呀，手动引用第三方类库呀，等布拉布拉一大堆问题。一个一个解决吧～)

原因猜测

使用CocoaPods管理第三方类库，在同步代码到远端时，只同步了Podfole和xxx.xcworkspace工程文件，真正存放第三方类库的Pods文件夹是被忽略的(可能是没必要把实际的文件同步，只要有Podfole文件就好啦，用到的时候在重新下载一遍喽！)，刚克隆的工程里木有Pods文件夹及第三方类库和Podfilelock文件。使用pod install命令后，根据Podfole文件重新让CocoaPods管理。
